\begin{abstract}
The divergence in in-memory index design has polarized between traditional cache-conscious structures (like CSB+-Trees) which suffer from dependent pointer-chasing, and modern Learned Indexes (like ALEX) which deliver high performance at the cost of massive, non-deterministic memory overheads. We introduce HydroDS, a highly concurrent, cache-conscious index structure utilizing contiguous memory buckets and SIMD instructions. HydroDS employs a novel \textit{Pressure-Flow} load-balancing algorithm and Optimistic Lock Coupling (OLC) to achieve deterministic performance. Our evaluation on bare-metal hardware reveals that HydroDS outperforms CSB+-Trees by 3x on point queries and 4.5x on range queries, while maintaining a memory footprint 2.75x smaller than ALEX. Furthermore, under high-concurrency, HydroDS exhibits a rare ``Concurrency-Induced Density'' property where aggressive multi-threaded insertions result in a tighter overall memory footprint. 
\end{abstract}

\section{Introduction}
Modern hardware architectures have reached a plateau in single-thread performance, forcing database systems to rely heavily on vertical multi-core scaling and dense CPU caches. While B+-Trees and their cache-conscious variants (such as CSB+-Trees~\cite{rao2000making}) minimize L2/L3 cache misses, they still suffer from pointer-chasing pipeline stalls. Conversely, recent Learned Indexes (like ALEX~\cite{ding2020alex}) eliminate pointer-chasing via Model-based array predictions. However, Learned Indexes mandate massive sparse arrays (gaps) to absorb writes, ballooning memory consumption to unacceptable levels in space-constrained environments.

To address this dichotomy, we present \textbf{HydroDS}, an in-memory concurrent index designed specifically for AVX2 vectorization, deterministic memory consumption, and extreme scalability. HydroDS abandons the standard pointer-heavy node architecture in favor of a flat array of highly-dense `Buckets' accessed via an interpolative directory. 

Our main contributions are:
\begin{itemize}
    \item \textbf{SIMD-Accelerated Buckets:} A cache-aligned flat bucket layout enabling AVX2 parallel keys matching, circumventing the need for deep binary searches.
    \item \textbf{Pressure-Flow Balancing:} A localized, vector-free reorganization algorithm that pushes elements into adjacent buckets instead of eagerly splitting, maintaining strict $O(1)$ node depths.
    \item \textbf{Optimistic Lock Coupling (OLC):} A lock-free traversal system coupled with a \texttt{std::shared\_mutex} directory lock that enables flawless linear multi-threaded scalability for reads and ranges.
    \item \textbf{Concurrency-Induced Density:} A demonstration of how parallel threads executing \textit{Pressure-Flow} cascades naturally compress the structure, actively lowering memory footprint under high concurrency.
\end{itemize}

\section{HydroDS Architecture}

\subsection{Cache-Aligned Contiguous Buckets}
Instead of nodes containing varying numbers of elements and pointers, HydroDS allocates uniform, cache-line aligned Buckets. Each Bucket (default Capacity $C=256$) contains a contiguous array of keys and payloads. When a query is routed to a Bucket, HydroDS uses \texttt{\_mm256\_cmpgt\_epi32} and \texttt{\_mm256\_movemask\_ps} (AVX2) to evaluate 8 keys per clock cycle. This contiguous layout allows the CPU's Hardware Prefetcher to load the entire bucket sequentially, rendering L1 cache misses non-blocking.

\subsection{Pressure-Flow Load Balancing (\texttt{stabilize})}
Standard B-Trees perform a split when a node is full, immediately allocating new nodes and decreasing average fill-factors to 50\%. HydroDS operates on a \textit{fluidic} principle. When a Bucket exceeds the high-watermark ($\epsilon_{high} = 0.85$), the \texttt{stabilize()} function shifts excess elements into adjacent buckets, provided they have available space. A bucket split is only triggered when the local neighborhood is entirely saturated.

\subsection{Concurrency and Lock Coupling}
To support multi-threaded operations, HydroDS utilizes Optimistic Lock Coupling~\cite{leis2019olc}. Each bucket contains an atomic version lock (Seqlock).
\begin{enumerate}
    \item \textbf{Directory Phase:} Readers acquire a \texttt{shared\_lock} on the global directory structure.
    \item \textbf{Bucket Phase:} Threads read the Bucket's version, execute the AVX2 search/insert, and verify the version post-execution. If a concurrent writer modified the bucket, the reader spins and retries.
    \item \textbf{Structural Phase:} Only structural modifications (splits) require upgrading to an exclusive directory lock.
\end{enumerate}

\section{Evaluation}
We evaluated HydroDS against a highly-optimized CSB+-Tree (from the \texttt{tlx} library) and ALEX (the state-of-the-art Learned Index). Benchmarks were executed on an Intel Core i5-12500H (12 Cores) with 16GB DDR4 RAM running Linux 6.8. All code was compiled using GCC 13 with \texttt{-O3 -mavx2 -fopenmp}. We populated the indexes with 5 Million dense 32-bit integers and executed mixed workloads.

\subsection{Single-Thread Latency and The Cache Paradox}
Table~\ref{tab:single_thread} highlights single-threaded operation latency. ALEX unsurprisingly dominates Inserts (2.33s) and Searches (0.039s) due to $O(1)$ predictive models. However, HydroDS drastically outperforms the CSB+-Tree across all metrics. Point Queries (Search) are 3x faster, and Range Queries are 4.5x faster (0.041s vs 0.185s) owing to sequential SIMD access.

Hardware profiling using \texttt{perf} revealed a paradox: HydroDS sustained \textbf{130.5 Million L1 Data-Cache misses}, whereas the CSB+-Tree recorded only 67.6 Million. Despite having 2x more L1 misses, HydroDS completed 3x faster. This validates that the \textit{pattern} of misses matters more than the volume; HydroDS's contiguous layout guarantees that misses are correctly predicted and pipelined by the CPU Hardware Prefetcher, whereas CSB+-Tree pointer-chasing results in pipeline stalls.

\subsection{Memory Footprint}
Learned Indexes trade space for time. As shown in Table~\ref{tab:single_thread}, ALEX consumes a massive 88 MB (18.45 bytes/key). In stark contrast, HydroDS uses only \textbf{32 MB} (6.71 bytes/key), proving that competitive latency can be achieved without the severe memory bloat associated with ML-based indexes.

\subsection{Multi-Threaded Scaling \& Density}
We scaled HydroDS from 1 to 32 concurrent threads using OpenMP (Table~\ref{tab:multi_thread}). Reads and range queries demonstrated excellent horizontal scaling, dropping to 0.011s for large scans. 

Crucially, as thread-count scaled, the total physical memory footprint \textit{decreased} from 33 MB to 28 MB. This \textbf{Concurrency-Induced Density} is a direct result of the \textit{Pressure-Flow} algorithm. When 32 threads simultaneously perform \texttt{stabilize()} operations across the tree, they create cascading compactions. Elements are pushed aggressively into adjacent buckets in parallel, raising the global bucket fill-factor and thus reducing total bucket allocations.

Write operations (Insert) exhibited negative scaling (2.53s to 5.21s). This is a known artifact of cache-line bouncing (contention) on the \texttt{std::shared\_mutex} reader-counter during the directory routing phase, leaving room for future lock-free improvements.

\section{Conclusion and Future Work}
HydroDS successfully bridges the gap between B-Trees and Learned Indexes. By combining contiguous SIMD buckets with \textit{Pressure-Flow} load balancing, it outpaces CSB+-Trees by 3-4x while maintaining a memory footprint 2.75x smaller than ALEX. Future work will replace the global directory \texttt{shared\_mutex} with Read-Copy-Update (RCU) or Epoch-Based Memory Reclamation to resolve write-contention and achieve true linear write-scaling.
